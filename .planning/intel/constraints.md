## CON-contract-artifact-openapi-json

- **source:** docs/ui_backend_contract.md (§ Contract artifacts 1)
- **type:** api-contract
- **content:** `openapi.json` (repo root) is the committed OpenAPI snapshot — the single
  source of truth for the API surface AND the device-state model shapes. Generated by the
  `wb-openapi` CLI (`wb_mqtt_bridge/cli/dump_openapi.py`) from the static route table
  (no MQTT/DB/hardware), dumping `app.openapi()`, deterministic (sorted, indented).
  Device-state models injected into `components.schemas` by
  `wb_mqtt_bridge/app/bootstrap.py::_install_openapi_with_state_models`. Injected set
  `OPENAPI_EXTRA_MODELS`: `BaseDeviceState`, the 7 device state models (`LgTvState`,
  `EmotivaXMC2State`, `KitchenHoodState`, `AppleTVState`, `AuralicDeviceState`,
  `WirenboardIRState`, `RevoxA77ReelToReelState`), `ScenarioWBConfig`, plus nested
  `LastCommand`. Regenerate + commit on any API or state-model change.
  (Implements DEC-contract-based-ui-backend-coupling, DEC-openapi-additive-state-model-injection.)

## CON-contract-artifact-device-state-mapping

- **source:** docs/ui_backend_contract.md (§ Contract artifacts 2)
- **type:** schema
- **content:** `config/device-state-mapping.json` maps each device class to its state
  model + config files; consumed by UI codegen. Paths (`deviceConfigs`,
  `scenarioConfigPath`) are relative to the mapping file's own directory, so the same
  file works in local-sibling and CI-subdir layouts. `stateClassImport` is
  `module:ClassName`; only `ClassName` is used today (looked up in `openapi.json`
  `components.schemas`); module path is vestigial.
  (Implements DEC-backend-owns-device-state-mapping.)

## CON-contract-artifact-device-configs

- **source:** docs/ui_backend_contract.md (§ Contract artifacts 3)
- **type:** schema
- **content:** `config/devices/*.json` per-device configuration (`BaseDeviceConfig`).
  Required fields: `device_id`, `device_name`, `device_class`, `config_class`; plus
  `device_category` (`device`|`appliance`, default `device`), `commands`, and optional
  WB-emulation fields. The UI reads these to build each device page; command order
  matters for layout.

## CON-build-time-codegen

- **source:** docs/ui_backend_contract.md (§ Build-time contract)
- **type:** protocol
- **content:** UI runs two generators against the sibling checkout:
  `gen:api-types` → `src/types/api.gen.ts` (`openapi-typescript` over `openapi.json`,
  REST request/response types); `gen:device-pages` → per-device `*.gen.tsx`,
  `*.hooks.ts`, `src/types/generated/*.state.ts` (state field shapes from
  `openapi.json components.schemas` via `src/lib/StateTypeGenerator.ts`; device pages
  from `config/devices/*.json`). Generated UI artifacts are gitignored (built fresh in
  CI); `src/types/api.gen.ts` is committed. No Python, no `pip install`.

## CON-runtime-rest-sse-mqtt

- **source:** docs/ui_backend_contract.md (§ Runtime contract)
- **type:** api-contract
- **content:** REST/SSE go through the UI's nginx proxy (`/api`, `/events`) to
  `BACKEND_HOST:BACKEND_PORT`; MQTT is a direct browser→broker WebSocket at `MQTT_URL`.
  REST: `POST /devices/{device_id}/action` (req `DeviceAction`
  `{action, params|null}`; resp `CommandResponse`
  `{success, device_id, action, state, error?, mqtt_command?, data?}`);
  `GET /devices/{device_id}/state`; config endpoints (`/config/devices`,
  `/config/device/{id}`, `/config/system`); groups; scenarios/rooms (`/scenario/*`,
  `/room/*`). SSE: `GET /events/{devices,scenarios,system,stats}` proxied with
  `proxy_buffering off` + long read timeout. MQTT: browser connects directly to broker
  WebSocket at `window.RUNTIME_CONFIG.MQTT_URL`.
  (Runtime URLs resolved per DEC-runtime-url-configuration.)

## CON-cross-repo-invariants

- **source:** docs/ui_backend_contract.md (§ Invariants — what MUST stay in sync)
- **type:** nfr
- **content:** Five load-bearing invariants:
  1. A device-state model must appear in `openapi.json`; add/rename → update
     `OPENAPI_EXTRA_MODELS` and `stateClassImport` ClassName, regenerate `openapi.json`,
     else UI codegen silently falls back to a basic state definition.
  2. `device_class` values in `config/devices/*.json` must match backend driver
     entry-points, UI device handlers (`src/lib/deviceHandlers/`), and the mapping keys.
  3. Regenerate + commit `openapi.json` on any API or state-model change (committed
     snapshot, not a running server, is what the UI build consumes).
  4. Command order in `config/devices/*.json` is load-bearing for layout (array-order
     zones follow command key order). Slated to become an explicit contract
     (see REQ-contract-based-button-placement / action_plan #10).
  5. The sibling checkout layout (`../wb-mqtt-bridge` local, `./wb-mqtt-bridge` CI/Docker)
     is assumed by the UI build; the mapping's relative paths depend on it.
